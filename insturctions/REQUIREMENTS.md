## 2. 프로젝트 목표

- 이 프로젝트가 해결하려는 문제: Journaling automation using OCR and sLLMs
- 사용자가 얻어야 하는 결과: 사진(영수증 등)을 업로드하면 화면에 분개 내용이 출력되는 간단한 분개 자동화 웹서비스
- 반드시 구현되어야 하는 핵심 기능: 
  - 필수 : 영수증 인식을 위한 OCR, 분개내용을 추론하기 위한 sLLM(ollama로 택)
  - 권장 : 백엔드 구현시 fastAPI로, SQLite를 사용해 분개 내용 띄우기, 구글 GCP를 통해 서버 구축
- 구현하지 않아도 되는 범위:
  - 거래인식 과정, 생각하는 과정, 유저 인터페이스(프론트엔드)에 사진업로드->업로드 된 사진 및 분개결과 를 제외한 것은 구현하지 말기, 프론트엔드 구현시 React등 프로젝트 규모에 비해 과한 라이브러리 사용 금지, 발표및 스크립트 등은 타 팀원이 맡을 예정

## 3. 회계원리 범위

- 다룰 회계 주제: 거래의 인식, 분개
  - 예: 거래의 이중성, 분개, 원장 전기, 시산표, 재무상태표, 손익계산서, 결산수정분개 등
- 사용할 계정과목 목록: 회계원리상 필수적인 계정과목 위주로 제한한다. sLLM이 임의 계정을 만들기보다 아래 계정 중 가장 가까운 계정을 선택하도록 한다.
  - Cash and Cash Equivalents
  - Bank Account
  - Credit Card Payable
  - Accounts Payable
  - Accounts Receivable
  - Sales Revenue
  - Sales Discounts
  - Employee Benefits Expense
  - Office Supplies Expense
  - Inventory
  - Cost of Goods Sold
  - VAT Receivable
  - VAT Payable
- 회계 기준 또는 수업 교재 기준: IFRS
- 예제 거래 수: <= 3
- 필요한 계산 방식: 차변 합계와 대변 합계가 반드시 일치해야 한다. 금액은 OCR 결과에서 추출한 숫자를 기준으로 하되, 불확실한 경우 sLLM이 임의로 금액을 만들지 않고 확인 필요 상태를 반환한다.

## 4. 사용자 시나리오

아래 시나리오는 발표 데모에서 우선적으로 검증할 사용 흐름이다.

### 시나리오 1

- 사용자: 김XX 사원
- 사용 목적: 사진에 대한 분개처리
- 입력값: 2,500원짜리 커피 2잔을 법인카드로 결제한 영수증 사진
- 처리 과정: 업로드된 사진을 서버에서 OCR로 인식 -> 추출된 텍스트를 ollama로 분개처리 추론 -> 추론 후 출력값(분개처리)를 sqlite로 db에 기록 후 -> db에서 프론트로 거래번호-계정과목-설명-차변금액-대변금액 순서 컬럼대로 출력
- 기대 결과: 김XX대리는 웹화면상 분개처리된 거래를 확인 

|No.|Date|Account|Description|Debit|Credit|
|---|---|---|---|---:|---:|
|1|YY-MM-DD|Employee Benefits Expense|Staff refreshments - coffee purchased with corporate credit card|5,000||
|||Credit Card Payable|Corporate credit card settlement payable||5,000|

(법인카드 결제이므로 현금 즉시 지급이 아니라 카드 미지급금으로 처리한다. 체크카드 또는 계좌 즉시출금 증빙이면 Credit Card Payable 대신 Bank Account를 사용할 수 있다.)

### 시나리오 2

- 사용자: 이XX 대리
- 사용 목적: 분개처리
- 입력값: 거래처가 물품을 최종 주문했고, 상품 발송 또는 인도 및 인보이스 발행이 완료되었음을 보여주는 신용장 또는 관련 거래 증빙 사진
- 처리 과정: 사진업로드 -> OCR 텍스트 추출 -> ollama 분개 추론 -> sqlite로 db 저장 -> db에서 프론트로 분개내역 출력
- 기대 결과: 
(시나리오1 이어서)

|No.|Date|Account|Description|Debit|Credit|
|---|---|---|---|---:|---:|
|2|YY-MM-DD|Accounts Receivable|Customer: AAA Ltd.; invoice amount after sales discount|6,500,000||
|||Sales Discounts|Sales discount granted on invoice|500,000||
|||Sales Revenue|Sale of product XXX; performance obligation satisfied||7,000,000|

(IFRS 관점에서 단순 주문 또는 신용장 개설만으로는 매출을 인식하지 않는다. 데모에서는 상품 인도 또는 수행의무 충족과 인보이스 발행이 확인된 경우에만 매출 분개를 출력한다. 매출원가와 재고 감소는 원가 정보가 입력된 경우에만 선택적으로 처리한다.)

## 6. 데이터 요구사항

- 입력 데이터 형식: 사진(png, jpg등)
- 저장해야 할 데이터:
  - 업로드 이미지 파일명 또는 저장 경로
  - OCR로 추출한 원문 텍스트
  - sLLM에 전달한 프롬프트
  - sLLM 원본 응답
  - 정규화된 분개 결과
  - 거래일자, 계정과목, 설명, 차변금액, 대변금액
  - 생성 일시 및 처리 상태
- 예시 데이터: 시나리오 1, 시나리오 2 참조
- 데이터 검증 규칙:
  - 파일 형식은 jpg, jpeg, png만 허용한다.
  - 파일 크기는 데모 환경 기준 10MB 이하로 제한한다.
  - OCR 텍스트가 비어 있으면 분개 추론을 진행하지 않고 오류를 표시한다.
  - sLLM 응답은 JSON 형태로 파싱 가능해야 한다.
  - 각 거래의 차변 합계와 대변 합계가 일치해야 한다.
  - 금액이 음수이거나 숫자로 해석되지 않으면 오류로 처리한다.
- 오류 처리 방식: 클라이언트에서 확인 가능한 한국어 에러 메시지를 표시한다. 서버 로그에는 원인 확인을 위한 상세 메시지를 남긴다.

## 11. 테스트 및 검증 기준

- 반드시 통과해야 하는 예제 입력:
  - 시나리오 1: 법인카드 커피 영수증 이미지 또는 동일 내용을 담은 테스트 이미지
  - 시나리오 2: 상품 인도 및 인보이스 발행이 확인되는 신용장/거래 증빙 이미지 또는 동일 내용을 담은 테스트 이미지
- 기대 출력:
  - OCR 텍스트가 화면에 표시된다.
  - 분개 결과가 No., Date, Account, Description, Debit, Credit 컬럼으로 표시된다.
  - 각 거래의 차변 합계와 대변 합계가 일치한다.
  - 결과가 SQLite에 저장되고 최근 분개 내역에서 다시 조회된다.
- 회계적으로 검산해야 할 항목:
  - 차변 합계 = 대변 합계
  - 비용 발생은 차변, 부채/수익 증가는 대변으로 처리되었는지
  - 법인카드 결제는 Credit Card Payable로 처리되었는지
  - 매출은 수행의무 충족 또는 상품 인도 조건이 있을 때만 인식되었는지
  - 매출할인은 Sales Discounts 또는 수익 차감 항목으로 일관되게 표시되었는지
- 오류 상황 테스트:
  - 지원하지 않는 파일 형식 업로드
  - OCR이 거의 불가능한 흐린 이미지 업로드
  - 금액 정보가 없는 이미지 업로드
  - Ollama 서버가 실행 중이지 않은 상황
  - sLLM 응답이 JSON 파싱에 실패하는 상황
- 익세미너가 중점적으로 확인해야 할 위험 요소:
  - sLLM이 근거 없이 계정과목이나 금액을 만들어내는 문제
  - 단순 주문을 매출로 잘못 인식하는 문제
  - OCR 오류가 회계 오류로 이어지는 문제
  - 차변/대변 불일치
  - 발표 시연 중 외부 환경 의존성 때문에 실행이 실패하는 문제

## 12. 제너레이터에게 전달할 개발 방향

- 제너레이터가 반드시 지켜야 할 구현 원칙:
  - 발표 데모에 필요한 최소 기능을 안정적으로 구현한다.
  - 프론트엔드는 단일 화면으로 구성하고 React 등 과한 프레임워크는 사용하지 않는다.
  - sLLM 응답은 반드시 구조화된 JSON으로 받도록 프롬프트를 설계한다.
  - JSON 파싱 실패, OCR 실패, Ollama 연결 실패에 대한 오류 처리를 구현한다.
  - 모든 자동 분개 결과는 차변 합계와 대변 합계를 검증한 뒤 저장한다.
- 제너레이터가 참고해야 할 회계 규칙:
  - 차변과 대변은 항상 균형을 이뤄야 한다.
  - 비용 증가는 차변, 부채 증가는 대변, 수익 증가는 대변으로 처리한다.
  - 법인카드 결제는 즉시 현금 감소가 아니라 Credit Card Payable 증가로 처리한다.
  - 매출은 단순 주문이 아니라 상품 인도 또는 수행의무 충족이 확인될 때 인식한다.
  - 불확실한 경우 sLLM은 확정 분개를 만들기보다 insufficient_information 상태를 반환한다.
- 코드 스타일 선호:
  - Python 기반 FastAPI 프로젝트
  - SQLite 사용
  - 단순한 디렉터리 구조
  - README에 설치, 실행, Ollama 모델 준비 방법 포함
- 우선 구현해야 할 기능 순서:
  - 1. FastAPI 서버와 단일 업로드 화면
  - 2. 이미지 업로드 및 저장
  - 3. OCR 텍스트 추출
  - 4. Ollama sLLM 호출 및 JSON 분개 생성
  - 5. 차변/대변 검증
  - 6. SQLite 저장 및 결과 테이블 출력
  - 7. 오류 처리 및 README 정리

## 13. 익세미너에게 전달할 평가 방향

- 익세미너가 중점적으로 검토해야 할 항목:
  - 요구명세서의 필수 기능이 실제로 구현되었는지
  - OCR 결과가 sLLM 분개 추론에 제대로 전달되는지
  - sLLM 출력이 구조화되어 있고 DB에 안정적으로 저장되는지
  - 회계 분개가 IFRS 관점의 기본 원칙에 부합하는지
  - 발표 시연 환경에서 재현 가능하게 실행되는지
- 회계 오류 판정 기준:
  - 차변 합계와 대변 합계가 불일치하면 중대 오류로 본다.
  - 거래 조건이 부족한데 매출, 비용, 자산, 부채를 확정 인식하면 오류로 본다.
  - 법인카드 결제를 근거 없이 Cash 또는 Bank 감소로 처리하면 개선 필요로 본다.
  - 계정과목은 영어 IFRS식 명칭을 우선 사용하되, 설명은 한국어 또는 영어 모두 허용한다.
- 코드 오류 판정 기준:
  - 실행 방법대로 실행되지 않으면 중대 오류로 본다.
  - OCR, Ollama, SQLite 중 하나라도 연결 실패 시 사용자에게 오류가 표시되지 않으면 오류로 본다.
  - sLLM 응답을 검증하지 않고 그대로 화면에 출력하거나 저장하면 위험 요소로 본다.
- 사용성 오류 판정 기준:
  - 업로드 후 처리 상태를 알 수 없거나 결과 테이블이 읽기 어렵다면 개선 필요로 본다.
  - 발표자가 데모 중 어떤 버튼을 눌러야 하는지 직관적으로 알 수 없으면 개선 필요로 본다.
- 개선 제안 방식:
  - 중대 오류, 일반 오류, 개선 제안으로 나누어 보고한다.
  - 각 항목은 문제 위치, 원인, 예상 영향, 수정 방향을 함께 제시한다.

## 15. 미정 사항

아직 결정되지 않은 사항을 적어 주세요.
